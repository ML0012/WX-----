# 微信支付回调服务 — 接入与对接说明

本文档面向负责「自建网站与本回调服务对接」的开发人员。

---

## 一、现状概览

本项目已实现的能力：

| 能力 | 状态 | 说明 |
|------|------|------|
| 接收微信支付异步回调 | 已实现 | `POST /api/pay/wechat/notify` |
| RSA-SHA256 签名验证 | 已实现 | 5 分钟防重放窗口 |
| AEAD_AES_256_GCM 解密 | 已实现 | 零额外依赖 |
| 幂等处理 | 已实现 | 已支付订单自动跳过 |
| 金额核对 | 已实现 | 校验应付金额与实际支付金额 |

**尚未实现、需要对接方补充的能力：**

1. **统一下单接口** — 调用微信 Native v3 API 生成 `code_url`（二维码链接）
2. **前端展示二维码** — 将 `code_url` 渲染为二维码供用户扫码
3. **真实订单数据库** — 当前为内存 Mock，重启数据丢失

## 二、回调地址（notify_url）

```
POST https://你的域名/api/pay/wechat/notify
```

**重要事项：**

- 此地址不是在微信商户后台全局设置的，而是在每次调用「统一下单 API」时通过请求体 `notify_url` 字段传递给微信
- 微信要求该地址必须 **HTTPS** 且 **公网可达**
- 本地联调可使用 ngrok / frp 等工具临时暴露到公网

## 三、部署要求

### 3.1 运行环境

- Node.js >= 16（推荐 18+）
- 依赖安装：`npm install`
- 启动：`npm start`（生产）或 `npm run dev`（开发热重载）

### 3.2 生产部署建议

```
[互联网] → [Nginx / 反向代理 (443 TLS)] → [本服务 (127.0.0.1:3000)]
```

- Nginx 配置 SSL 证书，将 `/api/pay/wechat/notify` 代理到本服务的 3000 端口
- 建议使用 PM2 / systemd 等保活进程管理器
- 健康检查端点：`GET /health` → `{"status":"ok"}`

### 3.3 环境变量

复制 `.env.example` 为 `.env` 填入真实商户参数：

| 变量 | 必填 | 说明 |
|------|------|------|
| `PORT` | 否 | 监听端口，默认 3000 |
| `WECHAT_MCHID` | 是 | 商户号（商户平台 → 账户中心） |
| `WECHAT_APIV3_KEY` | 是 | APIv3 密钥，**必须恰好 32 字节** |
| `WECHAT_PLATFORM_SERIAL` | 是 | 平台证书序列号或微信支付公钥 ID |
| `WECHAT_PLATFORM_PUBLIC_KEY` | 是 | 平台公钥 PEM，换行写为 `\n` |

> 详细填写指引见 `.env.example` 文件中的注释。

## 四、数据库对接（关键）

当前订单数据存储在 `src/repositories/orderRepository.js` 的内存 Map 中，仅用于演示。**上线前必须替换为真实数据库实现。**

### 4.1 需要实现的接口

只需实现两个函数，保持签名不变，Controller 层无需修改：

```javascript
/**
 * 根据商户订单号查询订单
 * @param {string} outTradeNo - 商户订单号（即下单时的 out_trade_no）
 * @returns {Promise<object|null>} 订单对象或 null
 *
 * 返回对象结构：
 * {
 *   outTradeNo: string,        // 商户订单号
 *   amountTotal: number,       // 订单应付金额，单位：分
 *   status: string,            // 'PENDING' | 'PAID' | 'CANCELLED'
 *   transactionId: string|null // 微信支付流水号（支付前为 null）
 * }
 */
async function findByOutTradeNo(outTradeNo) { ... }

/**
 * 将订单标记为已支付
 * @param {string} outTradeNo - 商户订单号
 * @param {string} transactionId - 微信支付流水号
 * @returns {Promise<boolean>} 更新是否成功
 */
async function markPaid(outTradeNo, transactionId) { ... }
```

### 4.2 关键约定

- **金额单位为「分」**：`amountTotal` 必须是整数，1 元 = 100 分。回调处理器会严格比对 `order.amountTotal === amount.total`（见 `src/controllers/wechatNotify.js` 第 85 行）
- **状态常量**：使用 `ORDER_STATUS.PENDING` / `ORDER_STATUS.PAID` / `ORDER_STATUS.CANCELLED`
- **幂等安全**：当 `status === 'PAID'` 时，回调处理器会直接返回成功，不会重复更新

### 4.3 替换方式

在 `src/repositories/orderRepository.js` 中替换 Mock 实现即可，导出保持不变：

```javascript
module.exports = {
  ORDER_STATUS,
  findByOutTradeNo,
  markPaid,
};
```

## 五、统一下单（待实现）

本项目不包含下单能力。对接方需自行实现调用微信 Native 支付下单 API。

### 5.1 接口信息

```
POST https://api.mch.weixin.qq.com/v3/pay/transactions/native
```

### 5.2 必要请求参数

```json
{
  "appid": "wxd678efh567hg6787",
  "mchid": "1900000000",
  "description": "商品描述",
  "out_trade_no": "YOUR_ORDER_NO_001",
  "notify_url": "https://你的域名/api/pay/wechat/notify",
  "amount": {
    "total": 100,
    "currency": "CNY"
  }
}
```

### 5.3 关键注意事项

- `notify_url` 必须填本回调服务的地址
- `out_trade_no` 要与你订单数据库中的记录一致，回调时以此关联
- `amount.total` 单位为分，必须与数据库中 `amountTotal` 一致
- 下单需要额外的凭证（本回调服务不需要这些）：
  - **商户 API 私钥**（`apiclient_key.pem`）— 用于请求签名
  - **商户 API 证书序列号** — 放在请求头 `Authorization` 中

### 5.4 下单成功响应

```json
{
  "code_url": "weixin://wxpay/bizpayurl/up?pr=NwY5Mz9&groupid=00"
}
```

将 `code_url` 生成二维码图片展示在网页上，用户扫码后完成支付，微信会回调本服务。

## 六、回调应答规范

微信对回调应答有严格要求，本服务已正确实现：

| 场景 | HTTP 状态码 | 响应体 |
|------|-------------|--------|
| 处理成功 | 200 | `{"code":"SUCCESS","message":"成功"}` |
| 处理失败 | 4xx / 5xx | `{"code":"FAIL","message":"错误描述"}` |

- 如果不返回 `200 + SUCCESS`，微信会按照 **15s / 15s / 30s / 3m / 10m / 20m / 30m / 30m / 30m / 60m / 3h / 3h / 3h / 6h / 6h** 的间隔持续重试，最长 24 小时
- 因此务必保证服务可用性

## 八、安全提示

1. `.env` 文件包含商户密钥等敏感信息，**绝对不能提交到 Git**（已在 `.gitignore` 中忽略）
2. `.pem` 私钥文件同样已在 `.gitignore` 中忽略
3. 生产环境建议通过环境变量注入或密钥管理服务（如 Vault）管理敏感配置
4. 确保回调服务仅对外暴露 `/api/pay/wechat/notify` 路径，其他管理接口不对公网开放
5. 定期轮换 APIv3 密钥，更新后同步修改 `.env` 并重启服务
